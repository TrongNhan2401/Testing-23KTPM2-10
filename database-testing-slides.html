Công cụ #1

migrate-mongo là gì?

what

Thư viện Node.js quản lý version hóa các thay đổi schema/dữ liệu trên MongoDB.

how

Mỗi thay đổi là một file migration với up() (áp dụng) và down() (rollback).

track

Lưu lịch sử migration đã chạy → dễ theo dõi, dễ khôi phục khi có lỗi.

role

Đảm bảo thay đổi dữ liệu được kiểm soát, nhất quán giữa các môi trường.

Ghi chú

Mỗi thay đổi schema/dữ liệu được viết thành một file migration gồm hai hàm: up() để áp dụng, down() để rollback khi có sự cố. Lịch sử các migration đã chạy được lưu lại, giúp kiểm soát chặt chẽ và nhất quán giữa các môi trường dev, staging, production.

Viết test invariant thủ công — lặp lại và dễ sai chỗ

no FK

MongoDB không có Foreign Key → mỗi collection (orders, products, users…) cần bộ test invariant riêng.

bug thật

Chạy standalone trên Windows: script im lặng bỏ qua , không có log lỗi nào — rất khó debug.

tốn công

Viết tay dễ quên field thật trong schema, hoặc đặt nhầm orphan-check ở collection sai.

scale

Càng thêm collection mới, càng tốn công viết lại logic kết nối DB, đọc .env , xử lý lỗi từ đầu.

Ghi chú

File test_invariants.js ban đầu được viết tay cho collection orders . Khi cần mở rộng sang products hay users , việc copy-paste thủ công dễ phát sinh lỗi: quên schema thật, đặt nhầm test tham chiếu sai file, hoặc gặp lỗi cross-platform (Windows) mà không có log báo.

migrate-mongo · pain point

Tại sao cần database migration cho MongoDB?

Thế giới SQL

structured
  • Có lệnh ALTER TABLE
  • Cấu trúc thay đổi đồng loạt, chặt chẽ

Thế giới NoSQL

schema-less
  • Document cũ dùng name
  • Document mới dùng fullName
  • Dữ liệu phân mảnh
nguy cơ

Khi deploy lên Production, code mới đọc dữ liệu cũ sẽ gây lỗi undefined và sập ứng dụng.

Ghi chú

Trong SQL, đổi tên một cột chỉ cần lệnh ALTER TABLE . MongoDB không có cấu trúc cố định: nếu đổi name thành fullName , dữ liệu cũ không tự cập nhật theo. Khi code mới được deploy lên Production, hệ thống có thể sập vì không đọc được dữ liệu cũ.

migrate-mongo · giải pháp

Giải pháp: migrate-mongo

tool

Công cụ quản lý phiên bản (database migration) dành riêng cho MongoDB trong môi trường Node.js.

↑ UP — nâng cấp

apply
  • Thực thi kịch bản chuyển đổi dữ liệu
  • Ví dụ: đổi toàn bộ name → fullName

↓ DOWN — rollback

revert
  • Đảo ngược quá trình an toàn nếu bị lỗi
  • Ví dụ: trả lại fullName → name
changelog

Tự động lưu vết các file đã chạy vào một collection changelog riêng trong DB.

Ghi chú

migrate-mongo hoạt động dựa trên 2 hàm chính: up() để nâng cấp cấu trúc, down() để rollback an toàn. Một collection changelog được tạo tự động để theo dõi script nào đã chạy, giúp việc deploy được kiểm soát và không lo mất dữ liệu.

Data masking · pain point

Data masking trong NoSQL — không chỉ thay thế chuỗi

nested

MongoDB chứa nhiều JSON lồng nhau (nested documents) như shippingAddress .

faker ok

Faker.js hoạt động rất tốt với các trường đơn giản (name, email…).

cần giữ

Địa chỉ lồng nhau cần giữ đúng cấu trúc JSON, đúng quốc gia, đúng định dạng mã bưu điện.

rủi ro

Mask thủ công dễ làm sai schema, tạo dữ liệu phi logic, gây lỗi khi kiểm thử tích hợp.

Ghi chú

Không phải mọi dữ liệu trong MongoDB đều là chuỗi đơn giản. Nếu chỉ thay ngẫu nhiên từng field bên trong shippingAddress , dữ liệu sinh ra rất dễ trở nên không hợp lệ — ví dụ mã bưu điện sai quốc gia hoặc thiếu key bắt buộc, làm hỏng cấu trúc JSON gốc.
